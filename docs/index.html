What is it

html-feedback is a small Claude Code skill that lets you review a folder of static HTML pages the way you'd review a Google Doc — by leaving comments directly on the rendered page — and have Claude edit the underlying source files in response. It serves your pages locally, injects a thin commenting layer into each one, and watches a tiny JSONL-based "mailbox" for things you want changed.

The core promise is simple: you stay in the browser, comments accumulate, click ▶ Process, Claude edits, repeat. There's no jumping back to the terminal between every change, no copy-pasting selectors, and no losing track of which suggestions have been applied. Every edit is snapshotted, walked through in the browser with a yellow highlight, and one-click revertible.

Use cases

Anywhere you'd otherwise be screenshotting a page, drawing arrows on it in Preview, and pasting the result back into a chat. The skill is happiest when the artifact is the HTML.

📊

Iterating on research artifacts

Tighten phrasing, fix captions, re-order sections in HTML reports without rebuilding from a notebook.

🤖

Reviewing AI-generated documents

When Claude generated the page in the first place, this gives you a clean follow-up loop to refine it without re-prompting from scratch.

📝

Copy-editing marketing pages

Highlight the awkward sentence, type "shorter, more punchy", click Process. The diff lands as a yellow highlight you can revert.

🎨

Design mockup reviews

Pure HTML/CSS prototypes — no full app needed. Comment on spacing, color, layout. Claude edits the styles in place.

📚

Documentation polish

Code-review-style edit cycles on docs sites or readme pages. Threaded replies mean you can push back on Claude's interpretation without losing context.

🗒️

Meeting notes & transcripts

Restructure, summarize, de-duplicate — annotate the rough draft and let Claude turn it into the clean version.

When to use vs when NOT to use

✓ Good fits

  • A handful to a few dozen static HTML files in one folder.
  • Iterative copy, structure, or styling changes.
  • You're the only person leaving feedback.
  • The page renders correctly when opened directly off disk (no required backend).
  • You'd rather see what you're changing than dictate it from the terminal.

✗ Not good fits

  • Live web apps with a backend, database, or auth state.
  • Hundreds of pages — the inject step touches every file, and the sidebar gets noisy fast.
  • Multiple people commenting at the same time (everyone runs their own local instance).
  • Anything that has to be reachable over a network — the server binds to 127.0.0.1 and won't change.
  • Files that aren't HTML (Markdown, PDFs, images — out of scope).

Quick start

From a clone of this repo:

git clone <repo>
cd html-feedback
python scripts/start.py --dir /path/to/your/html

That command:

  1. Adds feedback/ to your project's .gitignore (only if one already exists).
  2. Injects a tiny <link> + <script> tag into every *.html file (idempotently — re-runs are no-ops).
  3. Starts a local server on a random free port on 127.0.0.1, with a 32-char token in the URL.
  4. Opens your default browser to the first page.

Or, from inside Claude Code, in natural language:

start html feedback on ./pages

or use the slash command:

/html-feedback

Claude will run start.py for you, share the URL, and start watching feedback/control.jsonl for your ▶ Process clicks.

A full session typically looks like this:

1

Leave comments. Open the URL. Highlight any text and the selection toolbar pops up — click 💬 Comment, type your note. Or press e to enter element-pick mode and click a specific element. Or use the sidebar's + Page button for whole-page feedback.

Comments accumulate quietly in the sidebar. The 💬 floating button shows a red bubble with the unread count.

2

Click ▶ Process (N). When you've left as many comments as you want in this round, hit the blue button in the sidebar. That sends the whole batch to Claude.

A processing pill appears at the top of the page: Processing 1/4… updates in real time as Claude works through each comment.

3

Claude edits. For each comment, Claude takes a "before" snapshot, applies the smallest change that satisfies the request, takes an "after" snapshot, and writes a one-line summary to history.

4

Page auto-reloads. The Server-Sent Events stream notices the new history entries and the browser reloads. Every changed region is wrapped in a yellow highlight with Claude's summary as a tooltip.

Use j / k to step through the changes one at a time.

5

Revert anything you don't love. Each addressed comment in the sidebar carries a ↶ Revert link. Click it — or press r while the change is focused in the walkthrough. Patch-style: only that one edit gets undone, unrelated later edits stay put.

6

Redo if you change your mind. After a revert, the link flips to ↺ Redo. One click re-applies that exact edit.

7

Reply on a thread to push back. Even after a comment is addressed, you can hit Reply on it. A reply asking for further changes ("shorter still", "no, keep the old wording") goes into the next batch. Acknowledgements like "thanks" or "looks good" are silent no-ops — Claude won't try to interpret them as new work.

8

Clean up when done. Click clear addressed (N) in the sidebar header to sweep out finished comments. Start the next round with a clean slate.

Architecture

Three processes, three files. That's the whole picture:

+--------------------+        HTTP + SSE         +-------------------+
|  Browser           |  <----------------------> |  Python server    |
|  feedback.js/.css  |     127.0.0.1:<port>      |  lib/server.py    |
+--------------------+      ?t=<token>           +---------+---------+
                                                           |
                                                           | reads/writes
                                                           v
                                                 +-------------------+
                                                 |  feedback/        |
                                                 |   inbox.jsonl     |
                                                 |   control.jsonl   |
                                                 |   history.jsonl   |
                                                 |   session.json    |
                                                 |   .snapshots/     |
                                                 +---------+---------+
                                                           |
                                                           | tails control.jsonl
                                                           v
                                                 +-------------------+
                                                 |  Claude (Edit)    |
                                                 +-------------------+

The same picture, piece by piece:

🌐 Browser
feedback.js + feedback.css injected into each page
⇄
⚙️ Local server
Python stdlib only, 127.0.0.1, token-gated, idle shutdown
↕
📥 inbox.jsonl
your comments + soft-delete tombstones
▶ control.jsonl
"process" triggers from the UI
📜 history.jsonl
edits, reverts, external-edit markers
📸 .snapshots/
before + after HTML per edit (powers revert/redo)
↑ tail
🤖 Claude
reads control trigger, edits HTML, writes history

Only the browser writes to inbox.jsonl and control.jsonl (via the server). Only Claude writes to history.jsonl and edits *.html. The server watches both JSONL files and pushes Server-Sent Events to the browser so it can reload pages and run the change walkthrough. JSONL files are strictly append-only — soft-delete tombstones are how comments get removed.

In your target directory

PathPurpose
*.htmlYour pages. The inject step adds a small <!-- hfb:begin -->...<!-- hfb:end --> block; nothing else is changed.
feedback/inbox.jsonlAppend-only log of every comment you've made plus delete tombstones.
feedback/control.jsonlAppend-only log of ▶ Process triggers. Written by the server when you click; tailed by Claude.
feedback/history.jsonlAppend-only log of every edit, revert, and external-edit. Written by Claude.
feedback/session.jsonLive URL, token, PID, started-at. Overwritten on each start, deleted on clean shutdown.
feedback/.snapshots/<page>/Before-and-after HTML snapshots per edit. Powers patch-style revert and redo.

To wipe everything and start fresh: rm -rf <target>/feedback/. (On Windows: rmdir /s feedback.)

API reference

The endpoints the browser and Claude use. Every request needs a token via ?t=<token> or the X-Feedback-Token header — the server returns 403 otherwise.

MethodPathPurpose
POST/api/snapshotCopy the current page to .snapshots/<page>/<ts>.html and return the path. Called by Claude before and after every edit.
POST/api/feedbackAppend a new comment to inbox.jsonl.
POST/api/feedback/deleteSoft-delete a comment (and its replies) by appending a tombstone.
POST/api/feedback/clear-addressedSoft-delete every addressed comment on the page in one go.
POST/api/processAppend a {"type":"process","pending_ids":[...]} line to control.jsonl. This is what the ▶ button does.
POST/api/revertPatch-style undo of a specific edit, using the before/after snapshot pair.
POST/api/redoRe-apply a previously reverted edit by forward-applying the same diff.
GET/api/inboxReturns the inbox with each comment's status computed (open or addressed); tombstones filtered out.
GET/api/historyReturns every history entry.
GET/api/eventsSSE stream — emits history, inbox, and heartbeat events.

FAQ

Can I use this with backend-rendered pages?

No. The server serves files directly off disk — it has no idea how to run your backend. If your page needs a server to render correctly, save out a static snapshot first (Save As → "Webpage, complete" in most browsers) and point html-feedback at the snapshot folder. Comments on the snapshot won't propagate back to your backend templates automatically; you'll need to translate them yourself.

What if my page is huge?

The browser-side library is small and works on big pages, but very long documents are awkward to navigate in the sidebar. Two practical tips: (1) split the document into sections that are individually addressable, (2) use page-level comments for sweeping changes ("shorten everything") rather than highlighting half the page. Hundreds of HTML files in one folder is also rough — the inject step touches every page on startup.

Does it work offline?

Yes for the server, sidebar, and snapshots — all of those are local. Claude obviously still needs network access to do its thing, but the html-feedback infrastructure itself does not.

How do I share with my team?

You don't share the server — it binds to 127.0.0.1 only by design, and that's not negotiable. For a "team review" workflow, each person runs html-feedback locally on their own copy of the folder. Share comments by committing feedback/inbox.jsonl to git (it's append-only, so it merges cleanly most of the time) or by piping the edits via pull requests. This is fundamentally a single-user tool.

What's the difference between revert and redo?

Revert reverse-applies the diff of one edit — it undoes that specific change while leaving every other edit (before or after) untouched. Redo forward-applies the same diff, so you can re-do an edit you reverted. The same link in the sidebar flips between the two states.

Why didn't a highlight appear after Claude edited?

The walkthrough overlay searches for the after_snippet text inside the rendered page's document.body.innerText. If Claude only changed a style attribute, or changed something that isn't visible text, there's nothing to anchor a highlight to. The edit still happened — it's just not annotated. You can confirm by inspecting the file directly or checking history.jsonl.

What's an "orphan"? How do I re-pin?

An orphan is a comment whose anchor (the text or element it was attached to) no longer resolves in the current DOM — usually because the surrounding content was removed or rewritten in a later edit. Orphans land in a tray at the bottom of the sidebar with the original selected text shown. Click Re-pin, then select new text or click a new element to re-anchor the comment. The original thread is preserved.

Is anything sent to a cloud service?

No. The server, the inbox, the snapshots, and the history all live on your machine. The only outbound network traffic is whatever Claude Code itself sends to Anthropic when you run the skill — and that's the same traffic you'd have for any Claude Code session. The html-feedback layer adds nothing on top.

How do I clean up state?

Delete the feedback/ directory inside your target folder: 

rm -rf <target>/feedback/

That wipes all comments, history, snapshots, and the session file. The next run starts completely fresh. To also remove the injected <link>/<script> tags from your HTML files, run python scripts/inject.py remove --dir <target>.

Can I customize the colors / style?

Yes. The client CSS in lib/feedback.css uses CSS custom properties (all prefixed --hfb-) and every class is namespaced .hfb-. Override them from your page's own stylesheet, or edit feedback.css directly if you want global changes. The yellow change highlight is just background: rgba(255, 220, 100, .55) with a dashed border — easy to swap.

Can I run more than one of these at a time?

Sure — each run binds to a random free port and writes its own session.json into its own target directory. You'll get separate browser tabs with different tokens. Just don't point two instances at the same target directory.

The processing pill is stuck. What now?

The pill is driven by SSE. If your tab was suspended (laptop closed, background tab for a long time) the stream may have dropped. Refresh the page — the count is recomputed from the inbox/history on load, so you'll get an accurate state back immediately.

Security note

Local-only, token-gated, no network exposure.
  • The server binds to 127.0.0.1 only — never 0.0.0.0. Other machines on your LAN cannot reach it.
  • A 32-character URL-safe token is generated at startup. Every request (except GET /healthz) must carry it via ?t=<token> or the X-Feedback-Token header. Requests without it get a 403.
  • The token is never logged. Request logs go to stderr with method, path, and status only.
  • The server self-shuts down on idle (default 10 minutes of no authenticated activity) and when its parent process dies.
  • This is designed for single-user, local use. It is not hardened for shared servers, multi-user setups, or exposure beyond localhost.

Troubleshooting

Port already in use

The server asks the OS for a free port, so collisions are rare. If you do see a bind error, another instance is probably still running — check feedback/session.json for the PID and stop it, then restart.

Browser didn't open

Pass --no-open and copy the URL from the READY line start.py prints to your terminal. The URL includes the token query string and is the only way in.

Process button stuck on a stale count

The button shows the count of open root comments. After Claude edits, the count drops to 0 once the page reloads. If it doesn't, the SSE channel may have disconnected — refresh the page.

Comments don't appear in Claude's queue

Check that feedback/inbox.jsonl is actually being written to when you click Send. Then make sure you clicked ▶ Process — Claude only picks up comments when you trigger a batch, not automatically.

Page reloaded but nothing is highlighted

The walkthrough overlay looks for after_snippet text in the live DOM. If a history entry's after_snippet doesn't appear verbatim on the page (style-only changes, for example), no highlight will render. The change is still there — just not annotated.

Revert / Redo conflict

Patch-style revert requires the diffed hunk to still be present in the current file. If a later edit overlapped the same region, you'll get a conflict toast instead of a silent failure. Resolve by editing manually or by reverting the conflicting later edit first.

"Couldn't locate anchor" appears as a summary

This means Claude tried to address a comment but the text or element it pointed at was no longer findable in the source HTML (most often because the page was edited externally between the comment and Claude's read). The comment is marked addressed so it doesn't loop; re-pin it from the orphan tray to take another swing.

Server won't shut down cleanly

Ctrl-C in the terminal running start.py is the cleanest path. If that terminal is gone, open feedback/session.json, find the pid, and kill it directly. The server also self-shuts after 10 minutes of no authenticated activity, so worst case: leave it alone and it'll exit.
